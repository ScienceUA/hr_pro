# Архітектура та логіка системи HR-Pro (Cloud-Native & Stateful)

## 1. Призначення системи та Взаємодія з користувачем
HR-Pro — це ШІ-асистент для автоматизованого пошуку, збору та семантичного аналізу резюме кандидатів із різних джерел (Work.ua, Robota.ua, LinkedIn).
- **Інтерфейс:** Взаємодія відбувається через інтерфейс кастомного ChatGPT (Custom GPTs).
- **Головна мета:** Перетворити вільний текст вакансії на структурований пошук і видати відсортований, аргументований звіт.

## 2. Глобальна архітектура (Cloud Microservices & Stateful HITL)
Система логічно розділена на незалежні сервіси:

1. **HR-Pro Core (ШІ-ядро)**
   - Відповідає за логіку та семантичний аналіз.
   - **Stateful Workflow (Redis):** Використовує Redis для збереження стану сесій та результатів фонових завдань.
   - **Двокрокове API:**
     - **Endpoint `/preview`:** Синхронна розвідка. Отримує параметри пошуку, створює `session_id` та зберігає `SearchPayload` у Redis. Повертає кількість знайдених резюме.
     - **Endpoint `/analyze`:** Приймає `session_id`. Витягує параметри з Redis, запускає фонове завдання (`Background Task`) для парсингу та аналізу. Повертає `202 Accepted`.
     - **Endpoint `/status/{session_id}`:** Використовується для Long Polling, повертає поточний статус завдання та фінальний звіт.

2. **Parser Service (Мікросервіс збору)**
   - Ізольований модуль екстракції даних. Перекладає параметри пошуку в запити до сайтів та повертає уніфіковані дані `ParsingResult`.

## 3. Життєвий цикл запиту (Workflow) - Human-in-the-Loop
1. **Інтерпретація:** ШІ аналізує запит користувача та формує критерії пошуку.
2. **Розвідка (Preview):** Користувач бачить кількість потенційних кандидатів та `session_id`.
3. **Затвердження (HITL):** Користувач перевіряє критерії. Якщо все вірно, ініціює аналіз через `session_id`.
4. **Збір та Збереження:** Парсер збирає резюме. Кожен результат **обов'язково** зберігається у Raw Data Lake (GCS або локально в `out/`).
5. **AI-Скоринг та Фільтрація:**
   - Аналізатор оцінює кандидатів за статусами 🟢, 🟡, 🔴.
   - Усі кандидати зберігаються у Векторній БД (ChromaDB) для майбутньої фільтрації.
   - **RED Filter:** Кандидати зі статусом 🔴 (RED) ігноруються при генерації фінального звіту для користувача, але залишаються в базі даних.

## 4. Формат фінального звіту
Звіт містить лише релевантних кандидатів (🟢 та 🟡):
- **Ранжування:** Відсортовано за ступенем відповідності.
- **Структура картки:** Посилання на резюме, назва ролі та лаконічна аргументація (reasoning) щодо відповідності критеріям.

## 5. Надійність та Захист (Resilience)
- **Data Lake Resilience:** Сирі дані парсингу зберігаються до початку аналізу, що запобігає їх втраті.
- **Anti-bot:** Використання Jitter, Proxy Manager та Concurrency Limits.
- **Семантичне кешування:** ChromaDB запобігає повторному аналізу тих самих резюме.
