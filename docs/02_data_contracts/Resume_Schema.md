# Канонічна Схема Резюме (Resume Schema)

Цей документ описує єдиний стандарт даних (Pydantic-модель `Resume`), який використовується у всій системі HR-Pro. Будь-яке джерело (Work.ua, Robota.ua, LinkedIn) після парсингу зобов'язане повернути об'єкти саме в такому форматі.

## 1. Призначення Канонічної Моделі
- **Ізоляція ШІ від верстки:** Аналізатор (`analyzer.py`) нічого не знає про HTML чи специфіку API сайтів. Він працює виключно з чистими структурованими даними.
- **Економія токенів:** З резюме викидається реклама, навігація сайтів та важкі бінарні файли (base64).
- **Стійкість до відсутності даних:** Модель є максимально гнучкою. Більшість полів є опціональними, щоб парсер не падав, якщо кандидат не заповнив певні розділи (наприклад, освіту чи досвід). Відсіювання нерелевантних резюме виконується пізніше — на етапі ШІ-скорингу.

## 2. Структура Даних (`app/models/resume.py`)

### 2.1. Базова інформація (Resume)
Головний об'єкт профілю кандидата.
* **`id`** *(str, обов'язково)*: Унікальний ідентифікатор резюме на платформі (наприклад, "1234567").
* **`source`** *(str, обов'язково)*: Джерело походження ("workua", "robotaua").
* **`url`** *(str, обов'язково)*: Повне посилання на профіль для фінального звіту.
* **`title`** *(str, обов'язково)*: Вказана посада кандидата (наприклад, "Senior Python Developer"). Без цього поля резюме вважається невалідним.
* **`salary`** *(str, опціонально)*: Очікувана зарплата (якщо вказана).
* **`location`** *(str, опціонально)*: Місто проживання або готовність до релокейту/remote.
* **`skills`** *(List[str], за замовчуванням [])*: Масив ключових навичок (Tags).
* **`summary`** *(str, опціонально)*: Розділ "Про себе" (вільний текст) або автоматично витягнутий платформою текст із завантаженого файлу CV.

### 2.2. Досвід роботи (Experience)
Масив об'єктів `experience` (за замовчуванням `[]`).
* **`company`** *(str, опціонально)*: Назва компанії.
* **`role`** *(str, опціонально)*: Посада в компанії.
* **`period`** *(str, опціонально)*: Час роботи.
* **`description`** *(str, опціонально)*: Опис обов'язків та досягнень.

### 2.3. Освіта (Education)
Масив об'єктів `education` (за замовчуванням `[]`).
* **`institution`** *(str, опціонально)*: Назва навчального закладу.
* **`degree`** *(str, опціонально)*: Ступінь або спеціальність.
* **`year_end`** *(int, опціонально)*: Рік закінчення.

### 2.4. Знання мов (Language)
Масив об'єктів `languages` (за замовчуванням `[]`).
* **`name`** *(str, опціонально)*: Назва мови.
* **`level`** *(str, опціонально)*: Рівень володіння.

## 3. Правила Парсингу та Валідації
Мікросервіс збору (`parser_service/parsing/resume_parsers.py`) зобов'язаний дотримуватися таких правил:

1. **Ігнорування "Візиток" (Robota.ua):** Якщо профіль має тип `filling.type.id == "0"` (пуста візитка), він пропускається.
2. **Очищення сміття:** Усі HTML-теги повинні бути видалені або конвертовані у звичайний текст (Plain Text) до того, як дані потраплять у Pydantic-модель.
3. **Обробка вкладень (Robota.ua):** Поля у форматі `file.data` (завантажені PDF-файли, що віддаються як base64 або сирий бінарний код) наразі ігноруються для економії токенів, парситься лише заповнена анкета платформи, **або автоматично сформований джерелом текст із файлу** (його додаємо в поле `summary` або `description`).

## 4. Поточний Core Boundary (`CoreParsedResume`)

`app/models/parsed_resume.py::CoreParsedResume` є поточним контрактом межі між Core та результатами парсингу. Він використовується для передачі вже отриманого результату у Core/storage без прямої залежності storage від `parser_service.parsing.models.ParsingResult`.

Це не майбутня нормалізована бізнес-модель резюме. У межах цього контракту не потрібно додавати бізнес-поля на кшталт `hard_skills`, `experience_years` або інші поля, що належать до майбутньої нормалізації та екстракції.
