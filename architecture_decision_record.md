# Журнал архітектурних рішень (ADR)

Цей документ фіксує всі ключові архітектурні рішення, прийняті під час еволюції HR-Pro.

## ADR 001: Міграція до Cloud-Native мікросервісів

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
HR-Pro спочатку розроблявся як монолітний локальний CLI MVP. Оскільки ми плануємо масштабувати систему та надавати її через веб-інтерфейс (`upman.group`), система повинна бути розгорнута в хмарі, бути масштабованою та підтримувати одночасні сеанси користувачів, дотримуючись обмежень платформ (наприклад, виявлення ботів).

### Рішення
Ми мігруємо з локального MVP до архітектури мікросервісів на базі Google Cloud:
1. **Розділення сервісів:** Модулі `app/` (HR-Pro Core) та `parser_service/` (Parser) будуть розділені на два окремі Docker-контейнери. Core-сервіс було рефакторизовано з CLI-скрипта на FastAPI-додаток для підтримки синхронних/асинхронних веб-ендпоінтів.
2. **Ізоляція облікових записів:** `parser_service/` працюватиме на окремому обліковому записі Google Cloud для ізоляції проксі та активності збору даних від основного середовища Core-додатка.
3. **CI/CD:** Впровадження `cloudbuild.yaml` для перевірки лінтингу та тестів (TDD) перед розгортанням.

### Наслідки
- Потрібні API-контракти (REST) між сервісами Core та Parser.
- Необхідне надійне управління секретами (GCP Secret Manager) для Parser-сервісу.
- Додає складності при розгортанні, але значно покращує масштабованість, підтримуваність та безпеку.

## ADR 002: Глобальне сховище даних (Data Lake) та перевірка актуальності

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Резюме, зібрані з джерел (Work.ua, Robota.ua, LinkedIn), потребують часу, проксі та LLM-токенів для парсингу та векторизації. Виконання цього для кожного запиту користувача є неефективним. Крім того, резюме можуть ставати неактивними або прихованими з часом.

### Рішення
1. **Глобальне сховище:** ChromaDB та JSONL-сховище діятимуть як уніфікована глобальна база даних. Кандидати доступні всім сеансам користувачів.
2. **Валідатор актуальності:** Перед виконанням дорогого збору Core-сервіс запитуватиме векторний кеш. Для знайдених кандидатів Parser-сервіс виконуватиме швидкий HTTP HEAD/GET запит (Freshness Check), щоб перевірити, чи активна ще URL-адреса резюме.
3. Якщо URL активна (HTTP 200), вона повертається негайно. Якщо прихована/видалена (HTTP 404), вона видаляється з векторного кешу.

### Наслідки
- Різко знижує витрати на LLM-токени та затримку парсингу для частих запитів.
- Потребує створення `freshness_validator.py` всередині `parser_service/`.
- Вводить логіку синхронізації стану між Core та ChromaDB.

## ADR 003: Ізоляція облікових записів сервісу збору даних

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Збір даних із закритих платформ (наприклад, LinkedIn) вимагає авторизованих сесій (cookies/tokens). Використання облікових даних кінцевих користувачів є порушенням конфіденційності та безпеки, а також несе ризик блокування облікових записів користувачів.

### Рішення
1. **Виділені боти:** `parser_service/` використовуватиме пул виділених сервісних облікових записів ботів для автентифікації.
2. **Secret Manager:** Усі облікові дані ботів повинні зберігатися в Google Cloud Secret Manager. Жодних секретів у `.env` файлах репозиторію.
3. **Ротація:** Впровадження `AuthManager`, який безперешкодно ротує сесії та проксі, якщо обліковий запис бота буде заблоковано.

### Наслідки
- Запобігає блокуванню облікових записів користувачів.
- Потребує підтримки пулу валідних облікових записів ботів.
- Потребує інтеграції з GCP Secret Manager у `parser_service/`.

## ADR 004: Маршрутизація API Gateway

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Після розділення на мікросервіси нам потрібна єдина точка входу для клієнтів (наприклад, веб-інтерфейсу), яка абстрагує розташування та масштабування базових сервісів.

### Рішення
Впровадження Google Cloud API Gateway.
1. Шлюз використовуватиме специфікацію OpenAPI (`api_gateway.yaml`) для маршрутизації запитів.
2. Ендпоінти `/preview`, `/analyze` та `/status` будуть спрямовані до HR-Pro Core.
3. Внутрішні ендпоінти парсингу будуть спрямовані до Parser-сервісу на вторинному обліковому записі Google.

### Наслідки
- Централізована автентифікація, обмеження частоти запитів та маршрутизація.
- Веб-додаток взаємодіє лише з одним хостом API.

## ADR 005: Інфраструктура фронтенду

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Система HR-Pro потребує лендінгу та панелі керування для користувачів на домені `upman.group`.

### Рішення
Розміщення фронтенду як статичного SPA (Single Page Application).
1. Розгортання статичних активів (HTML/CSS/JS) у **Google Cloud Storage Bucket**.
2. Прив'язка бакета до Google Cloud HTTP(S) Load Balancer.
3. Налаштування Load Balancer із керованим SSL-сертифікатом для `upman.group`.

### Наслідки
- Надзвичайно низька вартість та висока доступність інтерфейсу.
- Швидка доставка контенту глобально, якщо увімкнено Cloud CDN.

## ADR 006: Управління станом (Memorystore та Persistent Volume)

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Контейнери Cloud Run є безстейтовими (stateless). Нам потрібен спосіб відстеження асинхронних завдань (статус завдань) та збереження наших векторних ембеддінгів.

### Рішення
1. **Транспортний стан (Redis):** Використання Google Cloud Memorystore (Redis) для зберігання `session_id`, кешування проміжних станів та зв'язку між асинхронним виконанням `/analyze` та ендпоінтами опитування `/status`. *Примітка: Під час розробки Фази 1 використовується тимчасовий словник у пам'яті.*
2. **Зберігання векторної БД (Persistent Volume):** Монтування тому Google Cloud Storage FUSE або Filestore як постійного диска в контейнер Cloud Run, де працює Core. Це гарантує, що файли ChromaDB SQLite та векторні дані переживуть перезапуск контейнера та діятимуть як Глобальне сховище даних (Data Lake).

### Наслідки
- Вирішує проблему стану для Cloud Run.
- Вводить залежності від сервісів стану, що потребують належного налаштування VPC для Memorystore.

## ADR 007: Redis для статусу завдань та збереження HITL

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Під час міграції Фази 1.5/2 до асинхронних фонових завдань FastAPI нам знадобився масштабований спосіб відстеження статусу завдань, оскільки початковий `TASK_STATUS` на базі `Dict` був несумісним із середовищами з кількома воркерами (і цільовою інфраструктурою Google Cloud Run). Крім того, потрібно було вирішити, як `criteria_bundle`, згенерований інтерпретатором, буде передаватися аналізатору у фоновому завданні.

### Рішення
1. **Сховище завдань Redis:** Ми негайно впроваджуємо Redis для управління станом завдань. Сервіс `redis` додано до `docker-compose.yaml` для емуляції Google Cloud Memorystore під час локальної розробки та тестування.
2. **Збереження HITL:** Інтерпретатор НЕ запускатиметься автоматично всередині `run_analysis_task`. Це б порушило логіку Human-in-the-Loop (HITL), яка передбачає, що користувач повинен переглянути та підтвердити критерії перед початком аналізу. Замість цього `criteria_bundle` передаватиметься через `SearchPayload` із фронтенду, гарантуючи, що користувач його вже схвалив.

### Наслідки
- Потребує наявності екземпляра Redis (локально через Docker або Cloud Memorystore у продакшені).
- Фронтенд відповідає за виклик ендпоінту інтерпретації, відображення результатів та вбудовування схваленого `criteria_bundle` у наступний POST-запит `/analyze`.

## ADR 008: Паттерн Стратегія для Cloud Storage та валідація актуальності

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
У Фазі 2 система повинна перейти до Глобального сховища даних на Google Cloud Storage, зберігаючи при цьому можливість працювати локально. Крім того, нам потрібно мінімізувати використання LLM-токенів та надлишковий парсинг, обробляючи лише *нові* або *змінені* резюме за допомогою нашого векторного кешу ChromaDB.

### Рішення
1. **Паттерн Стратегія для зберігання:** Ми вводимо інтерфейс `BaseRepository` з двома реалізаціями: `LocalStorage` (запис у JSONL) та `GCSStorage` (запис у Google Cloud Storage). Активна реалізація визначається змінною оточення `USE_GCS`.
2. **Валідатор актуальності:** Ми впроваджуємо робочий процес перед парсингом (`FreshnessValidator`). Перед парсингом система запитує ChromaDB на наявність семантично схожих резюме, перевіряє їх актуальність та розраховує дельту. Тільки відсутня частина буде зібрана адаптерами `workua`/`robotaua`.
3. **Ендпоінт внутрішнього очищення:** Процес очищення результатів перенесено з ізольованого скрипта до спеціального ендпоінту FastAPI (`/internal/cleanup`), захищеного заголовком `X-Cloud-Scheduler-Secret`.

### Наслідки
- Потребує бібліотеки `google-cloud-storage`.
- Локальна розробка залишається простою при `USE_GCS=false`.
- Робочий процес парсера суттєво змінюється: семантичний пошук та перевірка актуальності передують парсингу.

## ADR 009: Обов'язкове збереження сирих даних у Data Lake

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Оскільки ми переходимо до хмарної архітектури, спарсені резюме (сирі дані) становлять фундаментальну цінність системи. Втрата цих даних через помилки аналізу або збої контейнера є неприпустимою.

### Рішення
Ми встановлюємо правило, що кожен `ParsingResult`, отриманий від адаптера, ПОВИНЕН бути явно збережений у `BaseRepository` (LocalStorage або GCSStorage) оркестратором (`run_analysis_task`) ДО початку фази аналізу. Це гарантує стійкість Data Lake.

### Наслідки
- Логіка оркестратора стає дещо складнішою.
- Стійкість даних гарантована: навіть якщо аналіз LLM не вдасться, сирі дані надійно збережені в GCS/JSONL.

## ADR 010: Управління сесіями HITL на базі Redis

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Процес Human-in-the-Loop (HITL) вимагає, щоб користувач переглянув критерії, інтерпретовані з його запиту, перед початком глибокого аналізу. Багаторазова передача всього корисного навантаження з фронтенду є ризикованою та незручною.

### Рішення
1. **Збереження сесії:** Ендпоінт `/preview` генеруватиме `session_id` і зберігатиме `SearchPayload` (включаючи `criteria_bundle`) у Redis.
2. **Токенізований аналіз:** Ендпоінт `/analyze` прийматиме лише `session_id`. Він отримуватиме параметри пошуку з Redis, гарантуючи відповідність між тим, що бачив користувач, і тим, що аналізується.

### Наслідки
- Redis стає критичною залежністю для стейтфул-процесу.
- Логіка фронтенду спрощується: йому потрібно лише відстежувати та передавати `session_id`.

## ADR 011: Оцінка за Тріо-моделлю та фільтрація RED-кандидатів

**Дата:** 2026-05-01
**Статус:** Прийнято

### Контекст
Для підвищення якості звітів нам потрібна чітка система оцінювання, яка розрізняє ідеальні збіги, часткові збіги та некваліфікованих кандидатів. Крім того, фінальний звіт має містити лише актуальних кандидатів, щоб зменшити "шум" для користувача.

### Рішення
1. **Тріо-модель:** Ми впроваджуємо трирівневу систему оцінювання:
   - 🟢 **GREEN**: Відповідає всім обов'язковим та всім бажаним критеріям.
   - 🟡 **YELLOW**: Відповідає всім обов'язковим критеріям, але не має одного або кількох бажаних.
   - 🔴 **RED**: Не відповідає одному або кільком обов'язковим критеріям.
2. **RED-фільтр:** Оркестратор (`run_analysis_task`) автоматично відфільтровує кандидатів зі статусом 🔴 RED з фінального звіту в форматі Markdown, що надається користувачу.
3. **Персистентність:** RED-кандидати ВСЕ ОДНО зберігаються в Data Lake (GCS/JSONL) та векторному кеші (ChromaDB) для майбутніх запитів та запобігання повторному аналізу, але вони залишаються прихованими в основному звіті користувача.

### Наслідки
- Звіти стають більш фокусованими та "чистими" для рекрутерів.
- Система зберігає повну історію всіх проаналізованих кандидатів у фоні.
- Потребує узгодженого відображення статусів як в `ResumeAnalyzer` (інструкції LLM), так і в `ReportGenerator`.
