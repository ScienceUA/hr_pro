HR-Pro — Local MVP (Execution Layer)
Что это за проект

HR-Pro — учебный инженерный проект по созданию execution-сервиса для автоматизированного поиска и анализа резюме на Work.ua.

Текущая стадия проекта — Local MVP (L1.5).

На этом этапе:

сервис работает локально;

выполняет поиск резюме по заданному запросу;

загружает и парсит страницы резюме;

сохраняет результаты в локальный файл формата JSONL;

не использует прокси и облачную инфраструктуру.

Проект развивается поэтапно, с жёсткой фиксацией архитектурных решений на каждом этапе.

Текущий функционал (L1.5 — Local MVP)

На текущем этапе реализован полностью рабочий локальный краулер.

Он:

Принимает параметры поиска (query, city, pages).

Генерирует URL поисковой выдачи Work.ua.

Загружает SERP-страницы (списки резюме).

Извлекает ссылки на резюме.

Загружает страницы резюме.

Парсит структурированные данные (опыт, образование, навыки и т.д.).

Сохраняет результат в файл формата JSONL.

Останавливает выполнение при обнаружении BAN, CAPTCHA или LOGIN.

Это намеренно простой, синхронный и детерминированный MVP, предназначенный для:

отладки логики;

стабилизации контрактов данных;

подготовки к следующему этапу масштабирования.

Архитектурные решения (зафиксировано)
Decision A — эталон Local MVP

Для локального MVP эталонным транспортом является SmartFetcher.

Характеристики:

синхронный транспорт;

реализация на requests;

без прокси;

с явными задержками (throttling);

с простой и предсказуемой моделью ошибок.

Важно

Модуль Resilient HTTP (L1.3) реализован как подготовка к следующим этапам.

В runtime Local MVP он намеренно не используется.

Это осознанное архитектурное решение, а не ошибка и не технический долг.

Подключение httpx, tenacity, прокси и асинхронной обработки произойдёт на следующих этапах (L6 и далее).

На текущем этапе запрещено:

рефакторить Local MVP под async или httpx;

заменять SmartFetcher на другой транспорт;

«оптимизировать» код в ущерб прозрачности и воспроизводимости.

Запуск локального MVP (Smoke Test)

Для запуска требуется установленный Python 3.11 и Poetry.

Последовательность действий:

Установить зависимости проекта через Poetry.

Запустить main.py с параметрами поиска (query, city, pages, out).

Корректный результат выполнения:

в консоли выводится статистика выполнения (обработано страниц, сохранено резюме);

в указанном JSONL-файле появляются строки с JSON-объектами;

при бане или капче выполнение корректно завершается.

Smoke и диагностические тесты

В проекте присутствуют диагностические скрипты, подтверждающие корректность ключевых подсистем:

check_l2_2_logic.py — логика очередей и ограничений;

check_l2_2_net.py — сетевой стек;

check_l2_3_executor.py — retry и классификация ошибок (L1.3);

check_serp.py — парсинг SERP;

check_resume.py — парсинг резюме;

check_repository.py — сохранение и дедупликация данных.

Не все эти модули используются в runtime Local MVP, но они служат доказательной базой корректности реализации.

Структура репозитория (укрупнённо)

app
transport — SmartFetcher (эталон Local MVP)
parsing — базовый парсер, SERP и резюме, модели данных
services — оркестрация Local MVP и генерация URL
storage — локальное JSONL-хранилище
execution — модули L1.3 (Resilient HTTP, пока не в runtime)
core — базовые исключения и общие типы
